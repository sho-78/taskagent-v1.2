# TaskAgent — 草案・要件定義書 v0.5

> 起案: アーク
> 作成日: 2026-05-06

## 0. はじめに

本書は、外部提供を主眼として独立開発するタスク管理 SaaS の v0.5 草案である。3 回の方針確認、ローンチ戦略の詳細化、5 階層モデルの精緻化、UI/UX デザインガイドラインの確立、ビジュアルワークフロービルダーの導入を踏まえ、競合調査・要件定義・データモデル・ワークフロー・ロードマップを一括して提示する。

## 1. エグゼクティブサマリー

### プロダクトの一言定義

中小企業のビジネスチームから「タスクを考える時間」を取り除く、日本語ネイティブな AI エージェント型タスク管理 SaaS。

### 4 軸での差別化

| 軸 | 競合の現状 | 本プロダクトの方針 |
|---|---|---|
| 言語/UX | 英語ネイティブ or 日本語のみ翻訳 | 日本語業務文化を前提に設計 |
| 価格 | $19〜（個人）/$239〜（自動化） | 1 ユーザーあたり 1,000〜2,000 円帯 |
| 自律性 | スケジューラ or 拡張 AI | エージェント型（自律分解・優先度・通知） |
| 連携先 | Slack/Google 中心 | Google + Notion + LINE |

### MVP（v1.0）スコープ

- 階層管理（組織・部門・プロジェクト・タスク・サブタスク）
- AI 補助（タスク自動分解 / 優先度提案 / 議事録要約 → タスク化）
- 通知（Eメール + LINE 公式アカウント / アプリ内）
- KPI / 実績ダッシュボード（個人・部門・プロジェクト・組織の 4 軸）
- クロス組織コラボ（プロジェクト単位での別組織招待）

### ローンチ戦略（3 フェーズ）

| フェーズ | アクセス方式 | 目的 |
|---|---|---|
| フェーズ 1: 招待制クローズドベータ | 運営側が個別に組織を作成・招待 | プロダクト改善、初期事例化 |
| フェーズ 2: 招待拡大ベータ | 紹介・事前申込み | 口コミ機構の検証 |
| フェーズ 3: 外販 | Web 自己登録、フリーミアム | 本格的な顧客獲得 |

## 2. 競合調査

### 市場の 3 層構造

| レイヤ | 代表プレイヤー | 弱み（=狙い目） |
|---|---|---|
| AI-first スケジューラ | Motion / Reclaim / Clockwise | 英語 UI、個人軸、高価格 |
| 既存 PM + AI 拡張 | Notion AI / Asana / ClickUp Brain | 既存ユーザー前提、新規参入は不利 |
| 日本 SMB 向け | Jooto / スーツアップ / kintone | AI は拡張で主軸ではない |

### 戦略インプリケーション

- 「予定を自動配置する」だけでは Motion / Reclaim と差がつかない → タスクの分解・優先度・要約という上流の意思決定を肩代わりする
- Notion AI と kintone AI（2026 年 6 月正式版）が最大の脅威 → 専用ツールゆえの操作性と LINE 連携で防御線を張る
- 中小企業はツール数を増やしたくない → 既存の Google / Notion を置き換えず「司令塔」として機能する設計

## 3. ターゲット・ペルソナ

### 顧客セグメント

第一ターゲット: 従業員 10〜100 名規模の日本中小企業のビジネスチーム。

### ペルソナ A: 兼務マネージャー山田さん（38 歳）

- 営業マネージャー兼プレイヤー（30 名規模のホスピタリティ企業）
- ペイン: 自分のタスクと部下 5 名のタスクが混ざり、抜け漏れと特定個人への集中が起きる
- 求める価値: 「これだけ見ればわかる」状態と、考えなくてもチーム全体の遅延が事前に届くこと

### ペルソナ B: 現場リーダー佐藤さん（29 歳）

- 全国に小規模拠点を持つ運営会社の拠点リーダー
- ペイン: 本部からの指示と現場のイレギュラー対応が混在、報告に時間がかかる
- 求める価値: 音声 / LINE 経由でタスクが起票でき、終わったらワンタップで完了報告

### 非ターゲット

従業員 1,000 名以上の大企業 / エンジニアリング特化チーム / 完全な個人利用

## 4. プロダクト要件

### ビジョン

中小企業のビジネスチームのために、ToDo の整理・優先度判断・進捗監視を AI が代行することで、メンバーが「考えるべき仕事」に集中できる時間を作る。

### 提供価値

- 機能価値: タスク管理 + AI 自動分解 + 進捗ダッシュボード
- 情緒価値: 「考えなくていい」「忘れて大丈夫」という安心感
- 社会価値: 属人化の解消、心理的安全性の高いチーム運営

### テナント・ロールモデル

| ロール | 所属 | 主な権限 |
|---|---|---|
| プラットフォーム管理者 | 運営チーム | 全テナントの作成・停止・利用状況参照 |
| プラットフォームスタッフ | 運営チーム | サポート対応、利用状況閲覧（書き込み権限なし） |
| 組織管理者 | 顧客企業 | 自社テナント内のメンバー招待・権限管理・課金 |
| メンバー | 顧客企業 | 自社テナント内のタスク作成・編集・閲覧 |

## 6. データモデル

### 主要エンティティ

| エンティティ | 主な属性 |
|---|---|
| Organization | id, name, plan, status, provisioned_by |
| PlatformAdmin | id, email, role, last_login_at |
| Department | id, org_id, parent_department_id, name |
| Project | id, org_id, department_id, name, status, owner_id |
| Task | id, org_id, project_id, parent_task_id, title, status, priority, due_at, assignee_id, est_minutes, actual_minutes, source, ai_confidence |
| User | id, org_id, email, role, line_user_id |
| Membership | id, user_id, scope_type, scope_id, role |
| Invitation | id, org_id, email, token, expires_at |
| CrossOrgInvitation | id, host_org_id, guest_org_email, project_id, status, token |
| OrgRelation | id, host_org_id, guest_org_id, project_id, permissions |
| Tag, Activity | （省略） |
| AISession | id, org_id, kind, tokens, prompt, response |
| NotificationConfig | id, user_id, event_type, channel, enabled |
| NotificationLog | id, user_id, channel, payload, sent_at, read_at |
| LineLink | id, user_id, line_user_id, status |
| KpiSnapshot | id, org_id, scope_type, scope_id, period, date, metrics |
| Workflow, WorkflowNode, WorkflowEdge, WorkflowRun, WorkflowRunLog | v1.5 ワークフロー機能用 |

### 設計判断

- 全テーブルに `org_id` 必須、Supabase Row Level Security でテナント分離
- PlatformAdmin は User と完全に別テーブル
- 階層は柔軟で、Department / Project は任意
- サブタスクは parent_task_id 自己参照で N 階層
- KPI は事前集計（KpiSnapshot）してダッシュボード描画 1 秒以内
- ワークフロー状態は jsonb で永続化（React Flow が直接ロード）

実装は `supabase/migrations/0001_init.sql` を参照。

## 8. ワークフロー

### フロー 0: 組織オンボーディング（招待制ベータ）

運営 → 組織作成 → 初代管理者へ招待メール → アカウント作成 → メンバー招待

### フロー 1: AI タスク作成

ユーザー入力 → AI 分解 → 期日 / 担当推定 → プレビュー → 一括登録

### フロー 2: 進捗監視・通知

毎時バッチスキャン → リスク検知 → AI 言語化 → マルチチャネル配信

### フロー 3: 分析・改善ループ

週次集計 → AI 異常値検知 → マネージャー判断 → 翌週再計測

### フロー 4: 通知配信

イベント発生 → NotificationConfig 参照 → チャネル分岐（メール / LINE / アプリ内）→ ログ記録

### フロー 5: クロス組織招待

ホスト → 招待メール → ゲスト承認 → OrgRelation 確立 → 共有プロジェクト可視化

### フロー 6: ワークフロー実行（v1.5）

トリガー発火 → ノード逐次実行 → 条件分岐・AI・アクション → ログ記録 → 完了

## 11. KPI / 実績指標

### 経営 KPI（運営側）

ノーススター: WCAAT（Weekly Completed AI-Assisted Tasks）

AARRR は Acquisition 200 組織 / 月、Activation 70%、Retention Day30 50%、Revenue 8% 有料化、NPS 30、AI 出力人手修正率 30% 以下、粗利率 60% 以上を目標。

### 顧客向け実績 KPI（F-019 ダッシュボード）

4 軸（個人 / 部門 / プロジェクト / 組織）で各 5〜6 指標。実装方針:

- 日次バッチで KpiSnapshot に集計値を保存
- ダッシュボードは KpiSnapshot を読むだけ → 1 秒以内描画
- 期間: 今日 / 今週 / 今月 / 過去 7 日 / 過去 30 日 / カスタム
- CSV エクスポート全軸対応
- グラフは Recharts

## 12. UI/UX デザインガイドライン

### デザイン原則

1. 余白優先 — 詰め込まず視線の休息を作る
2. 階層の一貫性 — 色・タイポ・余白を統一
3. AI を主役にしない — AI 提案は控えめなアクセント、ユーザーが主体
4. 段階的開示 — 詳細は折りたたみ・モーダル
5. フィードバックの即時性 — 100ms 以内に反応

### ベンチマーク

参考: Linear / Notion / Vercel / Stripe / Tailwind UI

避けるべき: グラデーション、ガントチャート的詰め込み、複数アクセント、過剰アニメーション

### デザイントークン

カラー、タイポグラフィ、スペーシングは `apps/web/tailwind.config.js` と `apps/web/src/index.css` に実装済み。

主要トークン:

- プライマリ: `#2563EB`（AI アクセント `#7C3AED`）
- 8px グリッド
- フォントウェイトは 400 / 500 のみ
- 角丸: ボタン 6px / カード 8〜12px

## 14. セキュリティ要件

- 認証: メール + パスワード（Argon2id）、Google SSO、TOTP MFA（v1.5）

- データ保護: 保存時 AES-256、TLS 1.3、バックアップ 7 日
- 監査: Activity テーブルで全変更追跡
- AI 連携: テナント識別子匿名化、PII マスキング
- コンプライアンス目標: ISMS / プライバシーマーク（ローンチ後 1 年以内）
